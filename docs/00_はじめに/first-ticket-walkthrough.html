PractiCase 課題の進め方 — 最初の1本(T-001)完全ガイド

この文書は印刷するか、ブラウザで開いたままにして使います(印刷/PDF化: Ctrl+P)。
VS Code はコードの作業に集中してください。2本目以降の課題も Redmine から始めて同じ流れで進みます。
肩慣らしのチュートリアル課題(packs/php/tickets/01_チュートリアル/tutorial/)をまだやっていない人は、先にそちらを(30分で一周できます)。

パスの読み方: この文書に出てくるファイルパスは、すべてリポジトリの一番上(README.md がある階層)からの相対表記です。ファイルを探すときは VS Code で Ctrl+P を押してファイル名(例: features.md)を入力するのが最速です。見つからないときは、VS Code で開いているフォルダがリポジトリの一番上になっているか確認してください(下位フォルダを開くと検索範囲外になります)。

全課題共通の流れ

#やること使うもの
1Redmine で課題を選び、対応するチケットを読むRedmine / tickets/<課題>/ticket.md
2自分を担当者にし、見積をコメントして In Progress にするRedmine / retrospective テンプレの「見積」欄
3ブランチを切る(=着手の宣言)git switch -c feature/redmine-<issue番号>-<課題ID>-<短い名前>
4仕様書を読むsupport/spec.md → docs/01_設計資料/ の該当節
5症状を再現するブラウザ(アプリ)
6コードを読み、直すVS Code
7チェックを実行するdocker compose exec app php tools/check.php <課題ID>
8提出するPull Request(本文: docs/templates/fix_report.md)
9セルフレビューsupport/rubric.md(debrief がある課題は提出後に突き合わせ)
10振り返り → Closeddocs/templates/retrospective.md → reports/ / Redmine

fallback: Redmine が起動しない・接続できない場合だけ、ticket.md の front matter を open → in_progress → resolved → closed と更新します。通常は front matter を変更しません。Redmine と check.php の自動同期はないため、check の結果は自分でコメントに残します。起動・初期化・reset の詳細は Redmine 運用ガイドを参照してください。

T-001 の歩き方

手順1: Redmine からチケットを読む(約5分)

  • Redmine のプロジェクト「PractiCase Light」で、カスタムフィールド PractiCase Ticket ID = T-001 の issue を開く
  • 対応するファイル: packs/php/tickets/02_開発の基礎/T-001_job_validation/ticket.md
  • 4箇所に印を付ける: ①症状(3つ) ②期待動作 ③調査の入口 ④完了条件
  • チケットは「起きていること」と「あるべき姿」のペアで読みます。この差を埋めるのが作業です

手順2〜3: status 更新とブランチ

  • Redmine で自分を担当者に設定し、自分の見積(何分か+内訳)をコメントに残して、status を New → In Progress にする
  • docs/templates/retrospective.md を reports/T-001_retrospective.md にコピーし、「見積」の欄だけ先に書く(estimated_minutes を見る前に書くのがおすすめ — 完了後に答え合わせする)
  • ターミナルで git switch -c feature/redmine-<issue番号>-T-001-validation を実行する。ブランチ作成は「このチケットに着手した」という宣言です

手順4: 仕様書を読む(約10分)

  • support/spec.md が指す docs/01_設計資料/features.md の F-02 を開く
  • 「入力項目と検証ルール」の表を手元にチェックリストとして書き写す(項目/ルール/エラーメッセージ)
  • 仕様書は通読するものではなく、実装と突き合わせる道具です

手順5: 症状を再現する(約10分)

  1. docker compose --profile redmine up -d(未起動の場合)
  2. ブラウザで http://localhost:8180/login.php → クイックログインで「田淵(クライアント)」
  3. 「案件を登録する」でチケットの症状の値(時間単価にマイナスなど)を入力して登録する
  4. 登録が通ってしまうことを確認し、どの値が通ったかをメモする

修正の前に再現するのは、修正後に「直った」と判定する基準を作るためです。

手順6: コードを読み、直す(約30分)

  1. 画面の URL(/client/project_new.php)と public/ 配下のファイルが対応します
  2. packs/php/app/public/client/project_new.php を開く(入口。判断はしていない)
  3. その中で use / new されているクラスを辿ると、検証を担当するクラス(src/Service/ 内)に着きます
  4. 手順4の表と実装を1項目ずつ照合し、仕様にあって実装に無いチェックを特定する(症状の3つ以外の漏れも確認)
  5. 修正は、同じクラス内で正しく検証できている項目の書き方に合わせる。日時を扱う場合は coding-rules の ARC-5 を先に読む
  • 構造が分からなくなったら: docs/03_参考資料/code-tour.md(層の地図と処理の流れ)
  • 設計資料は必要な箇所だけ開く。この課題で主に見るのは docs/01_設計資料/features.md の F-02(テーブル定義は database.md、画面遷移は screens.md にあるが T-001 では通常不要)
  • 15分考えて進めなくなったら: tickets/02_開発の基礎/T-001_job_validation/support/hints.md を1段ずつ

手順7: チェック(修正の前後で2回)

docker compose exec app php tools/check.php T-001
  • 修正前に1回: 課題別テストの FAIL を確認(症状のテスト版)
  • 修正後に1回: PASS を確認。あわせて git diff で余計な変更が無いかを見る

手順8〜10: 提出・セルフレビュー・振り返り

  1. コミット → Pull Request を作成。具体的な手順は docs/02_作業ルール/git-and-pr-guide.md(push・PR画面・セルフマージまで)。本文は fix_report テンプレートで原因を自分の言葉で。check の PASS と提出物名を Redmine にコメントし、status を In Progress → Resolved にする
  2. support/rubric.md の観点でセルフレビュー
  3. 振り返り: reports フォルダを右クリック → 新しいファイル → T-001_retrospective.md を作り、retrospective テンプレートの項目を埋める。書き終えたら Redmine の status を Resolved → Closed にして完了

以降の課題への適用

  • 2本目以降も共通10ステップは同じ。変わるのは各課題フォルダの中身(症状・仕様の参照先・scope)だけ
  • 詰まったら各課題の support/hints.md(15分粘ってから1段ずつ)
