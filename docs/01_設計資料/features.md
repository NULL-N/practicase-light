# PractiCase 機能仕様書(MVP / Level 1〜3)

この文書は PractiCase(ITスポット案件マッチングシステム)の機能仕様を定める。
**学習者が業務チケットで参照する「仕様書」の正本**であり、実装・テスト・レビューの判断はすべてこの文書を根拠とする。

- 画面の一覧・権限は `docs/01_設計資料/screens.md`
- テーブル定義は `docs/01_設計資料/database.md`
- コーディング上のルールは `docs/02_作業ルール/coding-rules.md`

## 0. 横断仕様(全機能共通)

| # | 仕様 |
|---|---|
| G-1 | タイムゾーンは Asia/Tokyo 固定。日時の生成・比較はすべて PHP 側で行い、現在日時の取得は共通クラス Clock に一元化する(ARC-5)。テストでは基準日を固定して検証する |
| G-2 | 「締切日まで」は **締切日当日の 23:59:59 を含む**。翌日 00:00:00 から締切超過 |
| G-3 | 未ログインで要ログイン画面にアクセスした場合、S-01 ログインへリダイレクトする |
| G-4 | ロール外の画面・他社/他人のリソースへのアクセスは **404** を返す(存在自体を知らせない) |
| G-5 | 更新系(登録・応募・承認・却下・取り下げ・停止)はすべて POST + CSRF トークン必須。GET での状態変更は禁止 |
| G-6 | 更新成功後はリダイレクトし、結果をフラッシュメッセージで表示する(PRG: POST-Redirect-GET) |
| G-7 | バリデーションエラーは**全項目分をまとめて**表示し、入力値は保持する |
| G-8 | エラーメッセージ文言は本仕様書の記載を正とする(テストの期待値になる) |

## F-01 認証(ログイン/ログアウト)

### F-01 概要

email + password でログインし、セッションでログイン状態を保持する。

### ログイン(S-01)

- 入力: email(必須)、password(必須)
- 判定: users.email が一致し、`password_verify()` が成功し、status が `active` であること
- 成功時: `session_regenerate_id(true)` を実行し、ロール別トップへリダイレクト(engineer → S-03 / client → S-11 / admin → S-21)
- 失敗時(該当なし・パスワード不一致・suspended のいずれも同一文言):
  - 「メールアドレスまたはパスワードが正しくありません」
  - どの理由で失敗したかを画面から判別できないようにする(攻撃者への情報を減らす)

### ログアウト(S-02)

- POST のみ。セッションを破棄し S-01 へリダイレクトする。

### セッション仕様

- セッションには user_id のみ保存し、ユーザー情報は毎リクエスト DB から引く(停止反映を即時にするため)
- ログイン中に status が `suspended` になった場合、次のリクエストで強制ログアウトする

## F-02 案件登録(client)

### F-02 概要

client が自社の案件を登録する(S-12)。登録された案件は即座に `open`(公開中)になる。

### 入力項目と検証ルール

| 項目 | 必須 | ルール | エラーメッセージ |
|---|---|---|---|
| title(案件名) | ○ | 100文字以内 | 「案件名を入力してください」/「案件名は100文字以内で入力してください」 |
| description(案件内容) | ○ | 2000文字以内 | 「案件内容を入力してください」/「案件内容は2000文字以内で入力してください」 |
| hourly_rate(時間単価) | ○ | 1〜100000 の整数 | 「時間単価は1円以上10万円以下の整数で入力してください」 |
| capacity(募集人数) | ○ | 1〜100 の整数 | 「募集人数は1人以上100人以下の整数で入力してください」 |
| deadline(応募締切日) | ○ | YYYY-MM-DD 形式の実在日付、**本日以降** | 「応募締切日は本日以降の日付を指定してください」 |
| work_start_on(稼働開始日) | ○ | YYYY-MM-DD 形式の実在日付、**deadline 以降** | 「稼働開始日は応募締切日以降の日付を指定してください」 |
| is_remote(リモート可) | − | チェックボックス(未チェック=0 / チェック=1) | − |

検証の共通則:
- 数値項目は「整数表現の文字列」のみ受理する(空文字・小数・文字混じり・先頭ゼロ以外の記号は拒否)
- 形式エラー(YYYY-MM-DD でない・実在しない日付)は各項目のメッセージに含める(例: 締切日が `2026-02-30` → 「応募締切日は本日以降の日付を指定してください」)
- company_id はフォームから受け取らず、**ログイン中 client の所属企業を必ずサーバ側で設定する**

### 処理

1. CSRF・ロール(client)を確認
2. 全項目を検証(G-7: 一括表示)
3. projects へ INSERT(status = open)
4. S-11 へリダイレクト、「案件を登録しました」

## F-03 案件一覧・検索(engineer)

### F-03 概要

engineer が応募可能な案件を探す(S-03)。

### 表示対象(検索条件以前の母集合)

- status = `open` **かつ** deadline が本日以降(G-2: 当日を含む)
- 上記以外(draft / closed / 締切超過)は、検索条件に一致しても表示しない

### 検索条件

| 条件 | 仕様 |
|---|---|
| キーワード | title **または** description に部分一致(大文字小文字は区別しない)。空なら条件なし。LIKE 特殊文字(% _)はエスケープしてリテラル扱いする |
| リモート可のみ | チェック ON のとき is_remote = 1 のみに絞る。**OFF のときは絞り込まない(リモート可も不可も両方出す)** |

### 表示仕様

- 並び順: deadline 昇順(締切が近い順)、同日は id 降順
- 最大 50 件。0 件時は「条件に合う案件はありません」
- admin が S-03 を開いた場合は全件・全 status を表示する(検索条件は同様に適用)

## F-04 案件詳細(S-04)

- engineer: status = `open` の案件のみ閲覧可(draft / closed は 404)。締切超過でも open なら閲覧は可、ただし応募フォームは出さない(F-05)
- client: **自社案件のみ**閲覧可(他社は 404)。status 不問
- admin: 全件閲覧可
- 表示: 案件全項目+企業名+応募可否(engineer のみ)

## F-05 応募(engineer)

### F-05 概要

engineer が案件に応募する(S-04 内フォーム → S-05 POST)。

### 応募できる条件(すべて満たすこと)

| # | 条件 | 満たさない場合の文言 |
|---|---|---|
| A-1 | ログイン中ロールが engineer | (404) |
| A-2 | 案件の status が `open` | 「この案件は現在応募を受け付けていません」 |
| A-3 | 締切内(deadline 当日 23:59:59 まで) | 「この案件は応募を締め切りました」 |
| A-4 | 同一案件に応募履歴がない(**withdrawn も応募履歴に含む**) | 「この案件にはすでに応募しています」 |

- message は任意、500文字以内(超過時: 「応募メッセージは500文字以内で入力してください」)
- **募集人数(capacity)は応募の上限ではない**。応募は締切まで何件でも受け付け、capacity は承認数の上限として F-07 で判定する
- 成功時: applications へ INSERT(status = applied、applied_at = 現在時刻)、S-06 へリダイレクト、「応募しました」

### 取り下げ(S-06 から POST)

- 自分の応募かつ status = `applied` のときのみ可。status を `withdrawn` に、decided_at に現在時刻を記録
- 文言: 成功「応募を取り下げました」/ 対象が applied でない「この応募は取り下げできません」
- 取り下げ後の再応募は不可(A-4)

## F-06 応募者一覧(client / S-13)

- 自社案件のみ(project_id を書き換えた他社案件へのアクセスは 404)
- ヘッダ: 案件名/承認数/募集人数(例: 承認 2 / 募集 3)
- 一覧: **その案件の全応募**(applied / accepted / rejected / withdrawn すべて)を応募日時の新しい順に表示
- 状態の表示名: applied=「選考中」/ accepted=「承認済み」/ rejected=「却下」/ withdrawn=「取り下げ」

---

## F-07 承認・却下(client / S-14)

### F-07 概要

client が自社案件への応募を承認または却下する。

### 条件と処理

| # | 条件 | 満たさない場合 |
|---|---|---|
| D-1 | 対象応募が自社案件のものである | 404 |
| D-2 | 対象応募の status が `applied` | 「この応募はすでに確定しています」 |
| D-3 | (承認のみ)その案件の accepted 数 < capacity | 「募集人数の上限に達しているため承認できません」 |

- 承認: status = `accepted`、却下: status = `rejected`。いずれも decided_at = 現在時刻
- 成功文言: 「応募を承認しました」/「応募を却下しました」
- 承認数の判定は**処理時点の DB の accepted 数**で行う(画面表示時ではなく)
- 案件の status が `closed` でも、締切前に受けた applied の承認/却下は可能とする(掲載終了 ≠ 選考終了)

## F-08 権限制御(横断)

- 全ページ共通の前段で「ログイン確認 → ロール確認 → リソース所有確認」の順に検査する
- リソース所有確認の対象:
  - client: project / application が**自社(自分の company_id)のもの**であること
  - engineer: application(取り下げ・マイ応募)が**自分のもの**であること
- 所有違反・ロール違反・存在しない id はすべて 404 で統一(G-4)
- 判定は必ずサーバ側で行う。「ボタンを非表示にする」は UX 上の補助であり、権限制御とはみなさない

## F-09 ユーザー管理(admin / S-21)

- 全ユーザーの一覧(名前/email/ロール/状態)
- 停止: active → suspended。再開: suspended → active(いずれも POST)
- **admin は自分自身を停止できない**(「自分自身は停止できません」)
- suspended ユーザーはログイン不可(F-01)。ログイン中だった場合は次のリクエストで強制ログアウト

## 付録: シードデータ要件

動作確認・課題再現のため、初期データは以下を満たすこと(人物名・企業名は `docs/03_参考資料/world.md` に従う)。

| 内容 | 要件 |
|---|---|
| admin | 1名 |
| client | 2社・各1名 |
| engineer | 3名 |
| 案件 | 10件以上。open(締切前)/ open(締切超過)/ closed / リモート可 / リモート不可 / 定員1名 を最低1件ずつ含む。**課題の再現条件は先頭6件が担うため、先頭6件は変更しない(追加のみ)** |
| 応募 | applied / accepted / rejected / withdrawn を最低1件ずつ含む |

このバリエーションは、課題(検索・応募・承認・権限)の再現に直接使われる。
