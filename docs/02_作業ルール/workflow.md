# 学習の進め方(workflow)

PractiCase の課題は「問題」ではなく**業務チケット**として進める。
1本のチケットを、実務と同じ流れで最初から最後まで運ぶこと自体が訓練になる。

## 全体の流れ

```text
1. チケットを読む            packs/php/tickets/ の ticket.md(誰からの依頼かも読む)
2. status を更新し見積を書く  front matter の status を open → in_progress に。
                             同時に retrospective テンプレの「見積」欄へ自分の読み(分+内訳)を書く
3. ブランチを切る            git switch -c feature/T-001
4. 仕様書と既存コードを読む   support/spec.md → docs/01_設計資料/ → 対象コード。影響範囲を考えてから書く
5. 実装・作業する            チケットの scope の範囲内で
6. チェックを実行する        docker compose exec app php tools/check.php T-001
7. 提出する                  Pull Request を作る(本文は docs/templates/fix_report.md)
8. セルフレビューする        support/rubric.md の観点で自分の提出を見直す
9. 突き合わせる              debrief/ がある課題は、提出後に開いて自分との差分を言語化する
10. 振り返りを書く           docs/templates/retrospective.md → reports/ へ。status を closed に
```

## Issue 駆動(GitHub Issues との連携 — 任意だが推奨)

実務では、**Issue は作業の入口 / ブランチはその Issue への着手宣言 / PR はその Issue への回答**です。
3つは別々の操作ではなく、**1つの作業開始フロー**として繋がっています。この教材でも同じ流れを体験できます。

```text
Issue を読む → 担当を決める → Issue 用のブランチを切る(=着手の合図) → 作業 → PR で Issue に戻す
```

1. **Issues に課題を立てる**(どちらでも):
   - 推奨: `docker compose exec app php tools/issues-seed.php` が出力する gh コマンドを、
     自分のターミナルに貼って実行(gh CLI 導入済みの人。全課題が一括で立つ)
   - 代替: `... issues-seed.php manual` が出力する題名・本文を、GitHub 画面(Issues → New issue)にコピペ
     (gh が無くても大丈夫。教材は gh を必須にしません)
2. **着手 = ブランチを切る**: 取り組む Issue を決めたら、**Issue 番号と課題 ID を含めたブランチ**を切ります。
   これは単なる Git 操作ではなく「この Issue に私が着手した」という宣言です(命名は下のルール)。
   front matter の status も in_progress に
3. **提出**: PR 本文に **`Closes #番号`** と **`PractiCase: 課題ID`** を必ず書く(fix_report テンプレートの
   先頭に記入欄があります)。Closes / Refs の使い分け:
   - **`Closes #番号`** … この PR のマージで課題が**完了する**とき(修正・実装の PR)。マージと同時に Issue が自動で閉じる
   - **`Refs #番号`** … 関連するが**まだ完了しない**とき(例: 部分的な対応)。Issue は開いたまま残る

   これで「チケットを切る=Issue を作る」「ブランチを切る=着手する」「マージ=チケットを閉じる」が1本に連結します。

補足(識別子と正の所在):

- **課題の正式な識別子は PractiCase 課題ID(T-001 等)** — Issue 番号(#3 など)は人の環境ごとに変わるため、
  報告や会話では課題IDで呼ぶ。Issue の題名に `[T-001]` が入るのはこのため
- 教材の完了判定(check)が見るのは **front matter の status**(こちらが正)。
  Issues は「実務の工程管理の見え方」を体験する層で、二重管理になるぶん、閉じ忘れも実務らしさのうちです。

## ブランチとコミットのルール

- 作業は必ず main からブランチを切って行う。main 直コミットはしない
  - ブランチ名の**標準**: **`feature/issue-<Issue番号>-<課題ID>-<短い名前>`**
    (例: `feature/issue-3-T-001-validation`、`feature/issue-7-tutorial-label`)。
    Issue 番号が入ることで「どの Issue への着手か」がブランチ名だけで分かる
  - Issue を使わない場合の補助形式: `feature/<課題ID>-<短い名前>`(例: `feature/T-001-validation`)
- 1 Pull Request = 1 チケット(CHG-3)。コミットメッセージに課題 ID を含める(CHG-4)
- マージは自分のリポジトリなので自分で行ってよい(セルフマージ)。**マージの前に rubric のセルフレビューを終えること**

**git と Pull Request の具体的な手順(初めての人は必読)**: `docs/02_作業ルール/git-and-pr-guide.md`。
ブランチの切り方・コミット・push・GitHub での PR の作り方を、全手順つきで説明しています。
GitHub を使う方式(本物の PR)と、ローカルだけの方式(セルフマージ)の両方に対応しています。

## 複数の課題が同時に来たとき(トリアージ)

実務では、課題は1つずつ綺麗に届くとは限りません。**作業中に別の課題が割り込む**ことも、
**最初から複数を渡される**こともあります。そのとき最初にやるのは、コードを書くことではなく**順番を決めること**
(トリアージ = 選別)です。

- **`priority` を見る**: `high`(緊急・障害・顧客影響)は `normal` より先。ticket 冒頭の front matter にあります
- **title を見る**: 「障害」「エラー」「〜が壊れている」系は、たいてい先。ユーザーが今困っているからです
- **1つずつ・別ブランチで**: 割り込みが来ても、**今のブランチのまま別の課題を混ぜない**。
  片方をコミット(または提出)して区切りをつけ、**もう一方は別のブランチ**で扱います
  (1 PR = 1 課題。混ぜると、後でレビューも切り戻しもできなくなります)
- **迷ったら起票者に確認**: どちらを先にすべきか判断がつかないときは、それ自体を質問するのが実務です

> 複数の課題が同時に見えても、**教材の間違いではありません**。「どれから手をつけるか」もあなたの仕事です。

## check コマンド(正式)

```text
docker compose exec app php tools/check.php <課題ID>
```

- [1/3] 共通テスト … 既存機能を壊していないか
- [2/3] 課題別テスト … その課題の合格条件(無い課題は rubric でセルフチェック)
- [3/3] scope 検査 … 変更がチケットの範囲内か(未追跡の新規ファイルも見る)
- 設計の課題(D 系)の課題別テストは、コードではなく**提出文書・設計書の内容**を検査する
  (何を見るかは各チケットの完了条件に明記)

make が使える環境なら `make check T=T-001` でも同じことができる(補助)。

## 成果物の置き場所と status の運用(全課題共通)

### 報告書は reports/ に自分でファイルを作る

- VS Code のエクスプローラで `reports` フォルダを**右クリック → 「新しいファイル」**で作る。
  名前は `<課題ID>_<種類>.md`(例: `reports/T-000_setup_report.md`、`reports/T-001_retrospective.md`)
- 中身は `docs/templates/` の該当テンプレートを開き、コピーして埋める
- コードの変更はチケットの scope 内、報告書はすべて reports/ — この分担は全課題で同じ

### status は ticket.md の front matter を自分で直接編集する

- 各課題の `ticket.md` の**先頭にある `---` で囲まれた部分(front matter)**の `status:` 行を書き換えて保存する
- タイミング: 着手した時 `in_progress` → 提出した時 `resolved` → 振り返りまで終えた時 `closed`
- **チケットファイルは学習者が編集してよい**(status の運用は、編集することが前提の設計)。
  `reports/` と `packs/php/tickets/` 配下は、check の scope 検査で常に許可されている

## 推奨する進行順

進める順は1本道です。

```text
T-000 → tutorial → tutorial-2 → T-001 → T-012 → T-002 → T-013
→ D-010(要望整理) → D-011(基本設計) → D-012(DB設計) → D-013(詳細設計)
→ D-014(結合テスト+総合テスト仕様)
→ T-017(設計どおりに実装する)
→ T-014(SELECT) → T-015(GROUP BY/COUNT) → T-016(JOIN)
→ T-018(問い合わせ調査 — SQL とコード読解で原因を切り分ける)
→ T-019(コードを直す — ログイン不具合の修正)
→ T-028(他人の修正PRをレビューする — 回帰に気づく)
→ T-005(ここまでで「実務の一周」20課題が完了)
→ D-022 → D-023 → D-024 → D-025 → D-026 → D-027 → D-028 → D-029(設計基礎ドリル — 任意)
→ C-001 → C-002 → C-003(外部API入門 — 任意)
```

この並びには意図がある:
チュートリアル2本(「直す」を一周 → 「作る」を一周)で肩慣らしし、最初の本番 T-001(検証漏れの修正)へ。
**T-012 / T-013 は復習ドリル**(直前の課題と同じ型を、足場を減らして自力で解く)。

**D-010〜D-014 は、コードを書かない「設計の課題」**。お客様の曖昧な要望(1通のメール)を出発点に、
同じ1つの機能(案件タグ)を **要望整理 → 基本設計 → DB設計 → 詳細設計 → テスト仕様** の順で、
実装者に渡せる設計書まで育てる。実装の上流に必ずあるこの工程を、小さな題材で1周する。

**T-017 は、その設計を実装まで閉じる課題**。D-013 で自分が書いた詳細設計を、今度は実装者として
読み、コードに落とす。ただし範囲は Light 向けに絞ってある(検索の絞り込みだけ・タグを付ける画面は
対象外)。設計どおりにいかない部分が必ず出るので、retrospective でその食い違いを言語化する
(ズレを隠さず報告するのが実務)。

**T-014〜T-016・T-018・T-019・T-028 は「SQL と保守運用」の一続きの階段**。T-014 で **DB を自分の目で見る**
経験を作り(`SELECT`/`WHERE`/`ORDER BY`)、T-015(`GROUP BY`/`COUNT`)・T-016(`JOIN`)と技法を
積み上げ、T-018 で「実務の問い合わせに、SQL とコードの両方を読んで答える」という総合的な
調査課題に取り組む(コードは直さない・調査と報告のみ)。**T-019 では、今度は見つかった原因が
コード側の不具合だった場合に、実際にコードを直すところまで進む**(T-018 とは別の問い合わせが
題材)。**T-028 では立場が逆になり、別の人が書いた「T-019 と同じ不具合を直したPR」をレビューする**
——一見直っているPRが、実は別の条件を巻き込んで壊している、という実務で最も多い事故に、
コードを書かず気づく練習(この課題もコードは直さない・レビューと報告のみ)。

最後の T-005 で、再び手を動かして**「作る」(小さな仕様追加)**を経験して仕上げる。

**D-022〜D-029(06_設計基礎ドリル)は、最初の20課題で実務の一周を体験したあと、設計の判断力を
もう少し鍛えたい人向けの補強章**。「社内メンバー管理」という小さな題材で、テーブル分け → カラム →
型 → NULL/DEFAULT → UNIQUE → PK/FK → ER図 → DDL(CREATE TABLE)という個々の判断を1つずつ
素振りし、最後の D-029 では自分の CREATE TABLE 一式を自分でレビューして完成させる。
D-010〜D-014 が「設計の一連の流れ」を通す課題だとすれば、こちらは「個々の判断の筋力」を
つけるドリル。コードは書かない。取り組みは任意 — 一周を終えてから開けばよい。

**C-001〜C-003(07_外部API入門)は、自分のアプリの「外」にあるサービスとの連携を体験する
任意の補強章その2**。教材内で完結するローカルの通知基盤 PCP(架空のクラウドAPI)を相手に、
仕様を読み、API を呼び、応答と監査ログを確かめ、報告する — という外部 API 連携の基本動作を、
正常系(C-001)→ 認証エラー 401 の切り分け(C-002)→ 権限エラー 403 の切り分け(C-003)の
順で一周する。コードは書かず、提出物は各課題の報告書(reports/)。C-001 → C-002 → C-003 は
depends_on で連結しているので、この順で進める。

順序は強制しない(depends_on の連結は設計の4課題+T-017、SQL系のT-014→T-018→T-019→T-028、
外部API入門の C-001→C-002→C-003 のみ)が、
この階段を上る前提で足場(hints / code-reading)を配分している。

## CI(GitHub Actions)について

GitHub に自分のリポジトリとして置いている場合、push / Pull Request のたびに
`.github/workflows/check.yml` が自動で動き、アプリ起動+共通テストを実行する。

- 動くのは**あなた自身のリポジトリの無料枠**(public は無料。private も無料枠の範囲で足りる規模)
- secrets・外部サービス・有料 API は一切使わない
- CI の中身はローカルの正式コマンドと同じ。**CI が使えない環境(ZIP 運用・オフライン)でも、
  学習はローカルの check コマンドだけで完結する**
- 課題ごとの合格判定(check <課題ID>)はローカルが正式。CI は「既存を壊していないか」の見張り役

## 初めての人向けの足場(guide / hints / code-tour)

現場未経験でチケットの読み方から不安な人向けに、足場を用意している(使うかは自由):

- **`docs/03_参考資料/code-tour.md`** — コードの歩き方。どこに何があり、1操作がどう流れるか。最初に一度読むとよい
- **`docs/00_はじめに/first-ticket-walkthrough.md`** — 最初の1本(T-001)を最後まで運ぶ手順の完全版。印刷または
  別ウィンドウで開いたままにして、VS Code は作業に集中する(印刷用の同内容 HTML 版あり)
- 各課題の **`support/hints.md`** — 詰まったとき段階的に開くヒント(15分粘ってから1段ずつ)

足場は課題が進むほど薄くなる(T-001 は手厚く、後半は自走)。自力で進める人は使わなくてよい。

## 困ったとき

- 環境が壊れた: `docker compose exec app php tools/init-db.php` で DB を作り直せる。
  リポジトリごと壊れたら、配布物からやり直してよい(壊しても失うものはない)
- チケットの意図が分からない: support/spec.md と docs/01_設計資料/features.md の該当番号(F-xx / G-x)を読み直す。
  それでも曖昧な点は「仕様の確認事項」として報告書に書く(実務では質問すること自体が仕事)
