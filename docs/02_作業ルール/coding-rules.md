# PractiCase コーディング規約

この規約は PractiCase 開発チームの共通ルールである。
**コードレビューの指摘は、原則この規約のルール ID(例: SEC-1)を根拠として引用する。**
規約にないスタイルの好みは、レビュー指摘の対象にしない。

---

## ENV: 環境

| ID | ルール |
|---|---|
| ENV-1 | PHP 8.3 系に固定する |
| ENV-2 | すべての PHP ファイルの先頭で `declare(strict_types=1);` を宣言する |
| ENV-3 | タイムゾーンは Asia/Tokyo 固定(共通ブートストラップで設定し、個別ファイルで変更しない) |

## STY: スタイル(PSR-12 準拠の要点)

| ID | ルール |
|---|---|
| STY-1 | インデントは半角スペース4つ。タブは使わない |
| STY-2 | 1行は120文字以内を目安にする |
| STY-3 | 文字列は原則シングルクォート。変数展開が必要なときのみダブルクォート |
| STY-4 | 比較は厳密比較(`===` / `!==`)を使う。緩い比較(`==`)は理由をコメントで示せる場合のみ |

## NAME: 命名

| ID | ルール |
|---|---|
| NAME-1 | クラスは PascalCase(例: `ProjectRepository`) |
| NAME-2 | メソッド・変数は camelCase(例: `findOpenProjects`) |
| NAME-3 | DB のテーブル・カラムは snake_case(例: `hourly_rate`) |
| NAME-4 | 定数は UPPER_SNAKE_CASE |
| NAME-5 | 名前は役割が分かる語を選ぶ。`$data` `$tmp` `$flag` のような無情報な名前を避ける |

## ARC: 構成

| ID | ルール |
|---|---|
| ARC-1 | `public/` 配下のファイルは「入口」に徹する: リクエスト受付 → src のクラス呼び出し → 表示。業務ロジックを書かない |
| ARC-2 | SQL は Repository クラス(`src/Repository/`)に集約する。public/ 配下や テンプレート内に SQL を書かない |
| ARC-3 | 業務ルール(応募可否・承認可否など)は Service クラス(`src/Service/`)に置き、画面から独立してテストできる形にする |
| ARC-4 | 設定値(DB パス等)は 1 箇所の設定ファイルにまとめ、ハードコードを散らさない |
| ARC-5 | 現在日時の取得は共通クラス `Clock`(src/Support/Clock.php)に一元化する。業務コードで `date()` / `new DateTime('now')` / `time()` を直接呼ばない(Clock 内部が唯一の例外)。テストでは Clock の基準日固定機能を使い、実行日に依存しないテストにする |

## SEC: セキュリティ(最重要)

| ID | ルール |
|---|---|
| SEC-1 | SQL は必ずプリペアドステートメント(PDO のプレースホルダ)を使う。**変数の文字列連結・埋め込みで SQL を組み立てない** |
| SEC-2 | HTML への動的出力は必ず `htmlspecialchars($v, ENT_QUOTES, 'UTF-8')` を通す(共通ヘルパ `e()` を使う) |
| SEC-3 | パスワードは `password_hash()` で保存し `password_verify()` で照合する。平文・md5・sha1 での保存は禁止 |
| SEC-4 | ログイン成功時に `session_regenerate_id(true)` を実行する(セッション固定攻撃対策) |
| SEC-5 | 更新系 POST には CSRF トークンを必須とし、サーバ側で検証する |
| SEC-6 | 認可(ロール・リソース所有)の判定は必ずサーバ側で行う。リンクやボタンの出し分けだけで守ったことにしない |
| SEC-7 | ユーザー向けエラーには内部情報(SQL・スタックトレース・ファイルパス)を出さない |

## ERR: エラー処理

| ID | ルール |
|---|---|
| ERR-1 | `die()` / `exit()` によるエラー打ち切りを通常フローで使わない(リダイレクト直後の `exit` は除く) |
| ERR-2 | 予期しない例外は共通ハンドラで受け、ユーザーには一般文言、詳細はサーバ側ログに分離する |
| ERR-3 | 失敗しうる処理(DB・検証)の戻り値を無視しない |

## LOG: ログ

アプリの動作記録は `logs/app.log`(`App\Support\Logger`)。ERR-2 の「サーバ側ログ」の実体であり、
**障害調査の一次資料**になる。ログは「何かあったときに、条件を絞り込むための材料」— 出しすぎても
読めなくなり、出さなすぎても調査できない。

| ID | ルール |
|---|---|
| LOG-1 | パスワード・トークン・セッションID・Cookie の値は、どのレベルでも**絶対に出さない** |
| LOG-2 | 個人を特定する値は最小化する。利用者は **id のみ**(氏名・email は出さない)。ログイン失敗では入力された email も記録しない(入力は本人のものとは限らない) |
| LOG-3 | 1行1イベント。「日時 [レベル] イベント key=value ...」の形を守る(**grep で絞れる**ことが命) |
| LOG-4 | エラーには再現に必要な文脈(URL・パラメータ)を付ける — ただし LOG-1 / LOG-2 の範囲で。本教材では URL(検索キーワード等のクエリ込み)を記録しているが、**本番ではユーザー入力は個人情報を含む可能性があるため、記録は必要最小限にし、要否を設計時に判断する** |
| LOG-5 | ログの書き込み失敗でアプリを止めない(ログはベストエフォート) |

## FLOW: 制御フロー

| ID | ルール |
|---|---|
| FLOW-1 | ガード節(早期リターン)を優先し、ネストを浅く保つ。ネストは3段まで |
| FLOW-2 | 1メソッドは1つの仕事に絞る。目安として40行を超えたら分割を検討する |
| FLOW-3 | マジックナンバー・マジック文字列(status 値など)は定数化する |

## DOC: コメント

| ID | ルール |
|---|---|
| DOC-1 | コメントは「なぜそうするか」を書く。「何をしているか」の逐語コメントは書かない |
| DOC-2 | 仕様に由来する制約(例: 締切は当日23:59:59まで)はコメントで仕様書の該当項(F-xx / G-x)を参照する |

## CHG: 変更規律

| ID | ルール |
|---|---|
| CHG-1 | チケットの scope(front matter で指定された変更許可パス)の外を変更しない |
| CHG-2 | 修正と無関係な整形・リネーム・リファクタを同じ変更に混ぜない |
| CHG-3 | 1 つの Pull Request は 1 チケットに対応させる |
| CHG-4 | コミットメッセージにチケット ID を含める(例: `T-001: 案件登録の単価バリデーションを追加`) |

## TEST: テスト規律

| ID | ルール |
|---|---|
| TEST-1 | バグ修正では、まず**バグを再現するテスト**を書き、修正後に通ることを確認する |
| TEST-2 | 既存テストを削除・弱体化(assert の緩和・skip 化)してチェックを通さない |
| TEST-3 | 境界値(最小・最大・境界の前後)を最低1組は含める |

## 補足: DB 制約について(誤解防止)

本プロジェクトは入力値の検証をアプリケーション層に一元化し、DB の CHECK 制約を使っていない(docs/01_設計資料/database.md の D-4)。
これは「検証ルールとエラーメッセージを 1 箇所で読める」ことを優先した**本教材の設計判断**である。
実務の本番設計では、NOT NULL / UNIQUE / FK に加えて CHECK 制約など **DB 側の防御を重ねる多層防御が標準的な検討対象**になる。
「DB 制約は不要」という意味ではないことに注意すること。

## レビューでの使い方

- 指摘には根拠 ID を添える: 「SEC-1 違反: ここは文字列連結で SQL を組み立てています」
- 規約 ID で説明できない指摘は、仕様書(F-xx)またはチケット本文を根拠にする
- どちらでも説明できない「好み」の指摘は行わない

### レビューコメントの書き方(NullWorks 共通)

- 接頭辞で温度を示す: **[must]** 直してからマージ / **[ask]** 質問・確認 / **[good]** 良い点
- 「何が問題か → 根拠(規約ID・仕様番号)→ 期待する方向」の順で、コードに対して書く。人を主語にしない
- 良い点も最低1つ見つけて [good] で書く(レビューは攻撃ではなく品質を一緒に上げる作業)
