# Git と Pull Request の手順(初めての人向け)

各課題は「ブランチを切る → 直す → コミット → Pull Request → マージ」で進めます。
実務で毎日使う流れですが、初めてだと戸惑うところなので、ここに全手順を置きます。

まず**自分がどちらの方式で始めたか**を確認してください。方式でやり方が変わります。

| 方式 | Pull Request | どこで作業する |
|---|---|---|
| **A: GitHub(標準・この教材の前提)** | GitHub 上で本物の PR を作れる | ローカルで編集 → GitHub に push |
| **B: ZIP(オフライン時のみ・非推奨)** | GitHub が無いので PR は作れない。**セルフマージ**で代用 | ローカルだけで完結 |

**この教材は方式 A(GitHub)を前提とします。** 本物の PR・レビュー・CI の体験がこの教材の核だからです。
方式 B はオフライン等でどうしても GitHub を使えない場合の代替で、体験の一部が失われます。

---

## 共通(方式 A / B 両方): ブランチを切って直す

### 1. ブランチを切る

作業はいつも main から枝分かれさせます(実務で main を直接いじらないため)。

```text
git switch -c feature/issue-3-T-001-validation
```

`switch -c` は「新しいブランチを作って、そこに移動する」。名前は
**`feature/issue-<Issue番号>-<課題ID>-<短い名前>`**(Issue を使わない場合は `feature/<課題ID>-<短い名前>`)。
ブランチを切るのは単なる Git 操作ではなく、「**この Issue(チケット)に着手した**」という宣言です —
Issue が入口、ブランチが着手の合図、PR がその回答、と1本で繋がります(詳細: docs/02_作業ルール/workflow.md の「Issue 駆動」)。
今どのブランチにいるかは `git branch` で確認できます(`*` が今の位置)。

### 2. 直して、変更を確認する

コードを直したら、何を変えたか自分の目で見ます。

```text
git diff
```

赤(-)が消した行、緑(+)が足した行。**余計な変更が混じっていないか**をここで確認します。

### 3. コミットする(変更を1つの区切りとして記録)

```text
git add .
git commit -m "T-001: 案件登録の検証ルールを仕様どおりに実装"
```

- `git add .` = 変更したファイルを「次のコミットに含める」印を付ける
- `git commit -m "..."` = 印を付けたものをまとめて記録。メッセージには**課題ID を先頭に**(規約 CHG-4)

## 方式 A: GitHub で Pull Request を作る

### 4A. GitHub に push する

ローカルのブランチを GitHub に送ります。

```text
git push -u origin feature/T-001
```

`origin` = 自分の GitHub リポジトリ。`-u` は最初の1回だけ必要(2回目以降は `git push` だけでOK)。

### 5A. Pull Request を開く

1. ブラウザで自分の GitHub リポジトリを開く
2. push 直後なら黄色い帯に **「Compare & pull request」** ボタンが出るので押す
   (出ていなければ上部の **「Pull requests」タブ → 「New pull request」**、base=main / compare=feature/T-001 を選ぶ)
3. タイトルと本文を書く。本文は `docs/templates/fix_report.md` の項目を埋める(**原因を自分の言葉で**)
4. 緑の **「Create pull request」** を押す
5. これで PR 完成。CI(GitHub Actions)が動いていれば、テストの結果がこの画面に出ます

### 6A. レビューして、マージする(一人二役)

実務ではここで他の人がレビューします。この教材は一人なので、**自分がレビュアーの目で自分の PR を見ます**:

1. PR の「Files changed」タブで差分をもう一度見る(support/rubric.md の観点で)
2. debrief がある課題は、ここで開いて自分の提出と突き合わせる
3. 問題なければ **「Merge pull request」→「Confirm merge」** で main に取り込む
4. ローカルを最新にする: `git switch main` → `git pull`

---

## 方式 B: ローカルだけで完結する(GitHub 無し)

GitHub が無いので push と PR 画面はありません。代わりに:

### 4B. 差分を確認する(PR の「Files changed」に相当)

```text
git diff main
```

main と今のブランチの差分＝「もし PR を出すならこの内容」です。support/rubric.md の観点で見直します。

### 5B. セルフマージする

```text
git switch main
git merge --no-ff feature/T-001
```

`--no-ff` は「マージした履歴を1つの区切りとして残す」オプション。実務の PR マージに近い形になります。
マージの前に rubric と debrief を必ず終えること。

### あとで GitHub を使いたくなったら

方式 B で始めても、後から GitHub にリポジトリを作って push すれば方式 A に移れます。
その手順は setup-guide.md か、GitHub の「新しいリポジトリ作成」画面の案内に従ってください。

## コンフリクト(衝突)が起きたら

**まず落ち着いてください。コンフリクトは異常でも失敗でもありません。** チーム開発の**日常の一部**です。
2人が同じ場所を別々に直したとき、Git は「どちらを採るか自分では決められない」と言って、あなたに判断を委ねます。
それだけのことです。

### いちばん大事なこと(最初に覚える安全弁)

```text
git merge --abort
```

これは **「マージをやめて、衝突が起きる前の状態に戻す」** コマンドです。
**迷ったら、いつでもこれで元に戻れます。** 退路が常にあると分かっていれば、慌てる必要はありません。
「よし、まず戻せる。それから落ち着いて解決しよう」——この順番で構えます。

### ⛔ やってはいけないこと

- **`git reset --hard` は使わないでください。** これは**コミットしていない変更をすべて消す**強力すぎるコマンドで、
  コンフリクト解決の道具ではありません。取り返しがつかなくなります。**やり直したいときは必ず `git merge --abort`** を使います
- 衝突マーカーを**慌てて片方だけ消す**のもやめます(下記の「解決の手順」を落ち着いて踏みます)

### コンフリクトの見た目(3つの記号)

`git merge` で衝突すると、対象ファイルの中がこうなります:

```text
<<<<<<< HEAD
(あなたの変更)
=======
(相手=マージしようとしている側の変更)
>>>>>>> feature/mimura
```

- `<<<<<<< HEAD` から `=======` まで … **あなた**の変更
- `=======` から `>>>>>>> ...` まで … **相手**の変更
- この3つの記号(`<<<<<<<` `=======` `>>>>>>>`)は Git が入れた目印です。**解決したら必ず全部消します**

### 解決の手順

1. `git status` で、どのファイルが衝突しているか確認する(`both modified` と出る)
2. そのファイルを開き、3つの記号を探す
3. **どちらを残すか**を考える。多くの場合、正解は「**どちらか一方**」ではなく「**両方の意図を活かす**」です
   (相手も自分も、理由があってその変更をしています。片方を消すと、その人の仕事が消えます)
4. 記号(`<<<<<<<` `=======` `>>>>>>>`)を**すべて消し**、両方の変更が活きた正しい状態に書き直す
5. `git add ファイル名`(解決済みだと Git に伝える)
6. `git commit`(マージを完成させる。メッセージはそのままで良い)

途中で分からなくなったら **`git merge --abort` で最初からやり直せます**。何度でも。

## よく使う git コマンド早見表

| やりたいこと | コマンド |
|---|---|
| 今どのブランチにいるか | `git branch`(`*` が現在地) |
| 変更を確認 | `git diff` / `git status` |
| ブランチを切って移動(=着手宣言) | `git switch -c feature/issue-番号-課題ID-短い名前` |
| main に戻る | `git switch main` |
| 変更を記録 | `git add .` → `git commit -m "課題ID: 説明"` |
| GitHub に送る(方式A) | `git push -u origin feature/課題ID` |
| 間違えた、直前の変更を捨てたい | `git checkout -- ファイル名`(そのファイルを最後のコミットに戻す) |
| **マージをやめて衝突前に戻す(安全弁)** | **`git merge --abort`**(コンフリクトで迷ったら必ずこれ。`reset --hard` は使わない) |

## 用語

- **ブランチ**: 作業用に枝分かれさせたコードのライン。main を汚さず試せる
- **コミット**: 変更に区切りを付けて記録すること。1コミット=1つの意味のまとまり
- **push**: ローカルの記録を GitHub に送ること
- **Pull Request(PR)**: 「この変更を main に入れていいか」をレビューにかける依頼。実務のレビューの入口
- **マージ**: PR の内容を main に取り込むこと
- **コンフリクト(衝突)**: 2人が同じ場所を別々に変更し、Git が自動で1つにできない状態。異常ではなく日常。
  解決は「両方の意図を活かす」のが基本。迷ったら `git merge --abort` で戻せる(→「コンフリクトが起きたら」節)
