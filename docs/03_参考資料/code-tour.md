# コードの歩き方(code tour)

初めて PractiCase のコードを開く人へ。**どこに何があって、1つの操作がどう流れるか**だけを説明する。
(課題の解き方ではない。解き方は docs/00_はじめに/first-ticket-walkthrough.md と各チケットの `support/hints.md` にある)

> **パスの読み方**: この教材に出てくるファイルパスは、すべて**リポジトリの一番上(README.md がある階層)からの
> 相対表記**です。ファイルを探すときは、VS Code で **Ctrl+P** を押してファイル名(例: features.md)を
> 入力するのが最速です(実務でも標準の移動方法)。

## 大前提: アプリとコードとエディタの役割

| 場所 | 役割 |
|---|---|
| ブラウザのアプリ(localhost) | **症状を確認する場所**。バグを再現し、直った後に確かめる。ここでコードは直さない |
| エディタ(VS Code など) | **コードを読み・直す場所**。修正はすべてここで行う |
| `git diff` / Pull Request | **修正内容を見る・見せる場所**。何をどう変えたかはここで確認する |

「アプリの中でコードを直すのでは?」という誤解が一番多い。直すのはエディタ、確認はアプリ、見せるのは PR。この3つは別の場所だ。

## フォルダの地図

```text
packs/php/
  tickets/  ← 課題(業務チケット)。学習はここから始まる
  app/      ← アプリのコード本体(下記)
  sql/      ← SQL課題の提出ファイル(T-014)
```

```text
packs/php/app/
  public/     画面の入口。URL とファイルが対応する(/client/project_new.php = public/client/project_new.php)
  src/
    Service/     業務のルール(応募できるか・登録してよいか・検証)。「考える」層
    Repository/  DB への読み書き(SQL はここだけ)。「覚える/思い出す」層
    Support/     共通部品(ログイン・日時・CSRF など、どこからでも使う道具)
    Database.php DB 接続
  templates/  画面の共通パーツ(ヘッダ・フッタ)
  tests/      動作を保証するテスト。checks/ は課題ごとの合格条件
  database/   スキーマと初期データ
```

## 1つの操作がコードを流れる順番

「クライアントが案件を登録する」を例に、上から下へ流れる:

```text
[ブラウザ] 案件登録フォームで送信(POST)
    ↓
public/client/project_new.php   ← 入口。受け取って、下のクラスに渡すだけ(自分では判断しない)
    ↓
src/Service/(検証・業務ルール)  ← 「この入力は登録してよいか」を判断する。★T-001 の現場はここ
    ↓
src/Repository/(DBアクセス)     ← 判断を通ったデータを保存する
    ↓
[DB] 保存される → 画面に結果が返る
```

**入口(public)は受付、判断は Service、保存は Repository**。この役割分担(規約 ARC-1〜3)が分かると、
「入力が弾かれない」なら見るのは受付でも DB でもなく、**判断を担う Service** だと当たりが付く。

## T-001 で追う流れ

T-001 は「登録してはいけない値が登録できてしまう」課題。だから追うのはこの1本:

```text
public/client/project_new.php  →  src/Service/ の検証クラス
```

入口ファイルを開いて、その中で `use` されている・`new` されているクラスを辿れば、検証の現場に着く。
そこで `docs/01_設計資料/features.md` の F-02(検証ルールの表)と照合する — これが T-001 の中心作業だ。

## 用語ミニ辞書

- **Service(サービス)**: 業務の判断を書く場所。「単価は1円以上か」「締切は過ぎていないか」等のルール。
  画面から切り離してあるので、テストで単体検証できる
- **Repository(リポジトリ)**: DB とのやり取りだけを担当する場所。SQL を1箇所に集める(散らさない)ための層
- **なぜ層を分ける?**: 画面・判断・保存を混ぜると、どこを直せばいいか分からなくなる。
  分けてあると「入力チェックの話なら Service」と探す場所が決まる。これは実務の標準的な構え方
- **バリデーション**: 入力が正しいかの検証。不正な値を早い段階で弾く仕事
