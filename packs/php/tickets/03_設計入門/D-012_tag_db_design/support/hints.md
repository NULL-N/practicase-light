# D-012 hints(1段ずつ開く。15分粘ってから)

## 段1: 関係の判定は両方向から

- 1つの案件に、タグは何個付く? → 最大3つ = **複数**
- 1つのタグは、何件の案件に付く? → 制限なし = **複数**

**どちらの向きも複数 = 多対多**。多対多は2つのテーブルの間に「組み合わせを記録する表」
(中間テーブル)を置いて表現します。それが project_tags です。

## 段2: 「防ぐ」はどの道具で?

- タグ名の重複(「PHP」が2行できる)を防ぐ → `tags.name` に **UNIQUE**
- 同じ案件に同じタグを二重に付けるのを防ぐ → `project_tags` に **UNIQUE(project_id, tag_id)**

「最大3つ」はどうでしょう? — これは D-4(検証はアプリ層)により、**DB 制約では守りません**。
アプリ側の検証ルールとして D-013(詳細設計)に送ります。DB で守るもの・アプリで守るものの
仕分けが、この課題の核です。

## 段3: ER 図は「線の意味」だけ増やす

既存の図の書き方(`projects ||--o{ applications : "対象"`)を観察してください。
`projects ||--o{ project_tags` と `tags ||--o{ project_tags` の2本を足せば、
多対多が「1対多 × 2本」で表現できていることが図からも読めます。
