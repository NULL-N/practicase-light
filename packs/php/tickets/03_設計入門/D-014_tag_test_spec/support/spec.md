# D-014 spec: テスト仕様書に書く要素と型

## 確定済みの前提(D-010〜D-013までで決着済み)

- タグは運営が用意した固定の一覧(5つ)から、client が案件登録(S-12)時に **0〜3個** 付ける
- engineer は案件一覧(S-03)でタグを1つ選んで絞り込む(未選択なら絞らない)
- テーブル: `tags` / `project_tags`(多対多・複合ユニークで二重付与を防ぐ)
- 「最大3つ」はアプリ層で検証(D-4)

## 提出物の型(reports/D-014_test_spec.md)

この節立てをそのまま使ってください。

```markdown
# D-014 テスト仕様書:案件タグ(F-20)

## 対象範囲
(機能・画面・DB連携の確認+一連の業務シナリオの確認を、1枚にまとめて書く)

## テスト観点
| # | 観点 | 根拠 |
|---|---|---|

## テストケースと期待結果

### タグの登録(S-12・client)
| # | 操作 | 期待結果 |
|---|---|---|

### タグによる絞り込み(S-03・engineer)
| # | 操作 | 期待結果 |
|---|---|---|

### DB保存・参照確認
| # | 操作 | 期待結果 |
|---|---|---|

### 権限・異常系
| # | 操作 | 期待結果 |
|---|---|---|

### 境界値
| # | 操作 | 期待結果 |
|---|---|---|
| … | タグ0個で登録 | … |
| … | タグ3個で登録 | … |
| … | タグ4個で登録 | … |

### 業務シナリオ(通し)
(client がタグを付けて登録 → engineer がそのタグで検索 → 結果に出る、という一連の流れ)
```

## 観点ごとの中身(最低限これだけは)

| 観点 | 最低限含めること |
|---|---|
| タグの登録 | S-12 でのタグ選択・保存(project_tags への INSERT) |
| タグによる絞り込み | S-03 でのタグ選択・検索結果への反映。未選択時は絞り込まないことも |
| 表示 | S-04 で付与タグが表示されること |
| DB保存・参照 | project_tags への保存件数・絞り込み時の JOIN が正しいか |
| 権限 | 既存の権限マトリクス(S-03: engineer/admin、S-12: client)が変わっていないこと |
| 異常系 | 存在しない tag_id・タグ未指定の検索 |
| 境界値 | タグ **0個・3個・4個** |
| 業務シナリオ | 登録から検索結果への反映までの一連の流れ(最低1本) |

## 書き方の原則

- **実際に実行はしない**: このチケットは track: design です。「実施結果」「OK/NG」の列は書きません
- D-013(詳細設計)で決めたパラメータ名・エラー文言をそのまま使ってください。新しく発明しない
