# D-014 hints(1段ずつ開く。15分粘ってから)

## 段1: 観点は「今までの設計書」から拾う

新しく考える必要はほとんどありません。

- タグ登録・絞り込みの正常系 → D-011(基本設計)・D-013(詳細設計)がそのまま観点になる
- DB保存・参照 → D-012(DB設計)のテーブル定義がそのまま観点になる
- 権限 → screens.md の権限マトリクス(タグ機能で変わっていないこと)
- 境界値 → D-012 で決めた「最大3つ」という数字がそのまま観点になる

## 段2: 境界値は「ちょうど・1つ超え・1つ足りない」で考える

「最大3つ」という業務ルールから境界値を作るときの型です。

- ちょうど(3個): 上限どおりで成功するか
- 1つ超え(4個): 上限を超えて失敗するか
- もう一方の端(0個): タグ無しでも成功するか(タグは任意)

## 段3: 「結合」と「総合」を無理に分けない

このチケットは結合テストと総合テストを1枚にまとめます。ほとんどのケースは
「1つの画面・1つのDB操作」で完結する結合テスト寄りの内容でよいですが、
最後に1つだけ、**役割をまたぐ流れ**(client の登録が engineer の検索結果に
反映されるまで)を書いてください。これが「総合テスト」の視点です。
