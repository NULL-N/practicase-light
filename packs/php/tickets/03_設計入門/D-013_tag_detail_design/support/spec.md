# D-013 spec: 詳細設計書の型と決めるべきこと

## 提出物の型(reports/D-013_detail_design.md)

この節立てをそのまま使ってください。

```markdown
# D-013 詳細設計:案件タグ(F-20)

## 1. 画面と入出力
(画面ごとに、URL・メソッド(GET/POST)・パラメータ名・型・例 を表で)

## 2. 検証ルールとエラー文言
(F-02 の表と同じ型: 項目 / 必須 / ルール / エラーメッセージ)

## 3. 実装の割り当て(どの層に何を足すか)
(public のどのファイル / Service のどのクラス / Repository のどのクラスに何を足すか。
 新しいクラスを作るなら名前と責務)

## 4. 権限と異常系
(ロール外・存在しない id などのときにどうするか。既存の横断仕様 G-1〜G-8 を参照する)

## 5. テスト観点
(境界を最低2組。「何を入れたら、何が起きるべきか」の形で)
```

## この設計で決めるべきこと(チェックリスト)

- S-03(検索): タグ絞り込みのパラメータ名と値(タグの id か、名前か — 理由つきで選ぶ)。
  未指定・存在しない値のときの挙動
- S-12(登録): タグ入力のパラメータ名(複数選択をどう受けるか)。0〜3個の検証と、
  4個以上・存在しない id のときのエラー文言
- S-04(詳細): 表示のみ(入出力なし)で足りるか
- 検証はどの層でやるか(ARC-3)/ SQL はどこに書くか(ARC-2)
- タグ一覧の取得・案件との紐づけの保存を、それぞれ**どのクラスの何というメソッド**にするか

## 決めなくてよいこと

- 実際のコード(この課題では書かない)
- 画面の見た目・レイアウト(この教材ではデザインは扱わない)
