# D-013 hints(1段ずつ開く。15分粘ってから)

## 段1: 実装者の一日を想像する

実装者は、あなたの設計書を読んでこう動きます。

1. S-12 のフォームに選択肢を足す → **タグ一覧はどこから取る?**
2. POST を受けて検証する → **何個まで? 変な id が来たら? 文言は?**
3. 保存する → **どのテーブルに、どのクラス経由で?**
4. S-03 で絞る → **パラメータ名は? SQL の条件はどの層に書く?**

この4歩のそれぞれで「設計書に答えが書いてあるか」を確認してください。
1つでも「たぶんこうだろう」を実装者に残したら、そこが質問(=手戻り)になります。

## 段2: 迷いやすい2つの判断

- **絞り込みの値は id か名前か**: URL に出るもの(`?tag=2` か `?tag=PHP` か)。
  どちらでも設計できますが、「タグ名は将来変わりうるか」「存在チェックのしやすさ」で
  理由をつけて選んでください。選んだ理由を書くところまでが設計です
- **「最大3つ」はどこで守る?**: D-012 で「DB では守らない(D-4)」と決めました。
  なら Service の検証です。エラー文言も、既存の F-02 の文体(「〜で入力してください」)に
  そろえて自分で確定させます

## 段3: テスト観点は「境界の前後」をペアで

「3個まで OK」なら、観点は最低このペアです。

- 3個 → 登録できる
- 4個 → エラー(文言まで一致)

同じ型で「0個(タグなし)」「存在しない id」「タグ未指定の検索」「該当0件の検索」を
並べれば、5. の節はほぼ埋まります。
