# C-002 期待される報告の観点(答え合わせ用 — 提出後に読む)

## この課題の狙い

認証エラーは、実務では「たまに動かない」という曖昧な問い合わせの形でやってくる。
**わざと失敗させて、失敗の形を先に知っておく**と、問い合わせを受けたときに
「401ですか? それなら認証情報の問題です。ログの suffix は残っていますか?」と
一直線に切り分けられる。

## 報告書に入っていてほしい要素

### 何を試したか

- 成功1(正しいキー・201)+失敗2(キー無し・誤ったキー、どちらも401)の**対照実験**
- 「誤ったキー」にも明らかなダミー(`PCP_TEST_KEY_0000000000000000`)を使ったこと
  (実在サービスのキーらしき文字列を実験に使わない、という習慣そのものが安全上の型)

### 401から何が分かったか

- エラーコードは `INVALID_API_KEY` で、キー無しでも誤ったキーでも**同じ401**が返ること。
  サーバーは応答では「キーが無い」と「キーが違う」を区別して教えない —
  呼び出し側(攻撃者かもしれない)に余計なヒントを渡さない、一般的な設計
- 401 = 認証の失敗 = サーバーがこちらを「誰か分からない」と言っている状態
- **再送では直らない**。リトライではなく、キーとヘッダーを直すのが対処

### 監査ログに何が残ったか

- 応答では区別されなかった2つの失敗が、**ログでは区別できる**:
  キー無し → `api_key_suffix` が null / 誤ったキー → 末尾4桁が残る
- つまり「外(応答)には最小限、内(ログ)には切り分けに足る情報」という**非対称**が
  意図された設計であること。運用側はログで原因を即答でき、外部にはヒントを漏らさない

## よくある不足

- 成功系(201)を試さない — 失敗だけ見ても「正しければ通る」ことの確認にならない。
  **対照の無い実験は、報告として弱い**
- 「401が出た」で止まる — 401 が**何を意味するか**(誰か分からない・再送では直らない)
  まで書いて、初めて切り分け報告になる
- 401 と 403 を同じ「権限エラー」と括ってしまう — 認証と認可は別物。
