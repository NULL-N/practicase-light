# C-002 ヒント(詰まったら上から順に)

## ヒント1: どれを実行しても 201 になる(401 が出ない)

「キー無し」のつもりで `-H "Authorization: ..."` を消し忘れていないか確認する。
`support/spec.md` の失敗1の例は、Authorization ヘッダーの行ごと存在しない形になっている。

## ヒント2: どれを実行しても 401 になる

正しいキーの綴りを確認する。ありがちなのは、`Bearer` の後の半角スペース抜け・
キーのコピペ時の欠け・`Bearer` の綴り間違い。
`support/spec.md` の成功例をそのままコピーして実行するのが確実。

## ヒント3: audit-log の絞り込みが効かない

URL 全体を引用符で囲んでいるか確認する。`?` や `&` はシェルに解釈されるため、
`"http://pcp:8080/v1/audit-log?event_type=INVALID_API_KEY"` のように囲む。

## ヒント4: api_key_suffix が null の意味が分からない

null は「キーとして解釈できるものが、そもそも送られてこなかった」という意味。
ヘッダー無し・`Bearer` 形式になっていない場合がこれにあたる。
一方、形式として正しいキーが送られてきた場合は、登録されているかどうかに関係なく
**末尾4桁だけ**が記録される(全文は決して記録されない — C-001 で見たとおり)。

## ヒント5: 報告書に何を書けばいいか

3パターン(成功・キー無し・誤ったキー)を小さな表にして、
「HTTPステータス / error code / api_key_suffix」を並べると、
切り分け報告として一目で伝わる形になる。
