# C-003 期待される報告の観点(答え合わせ用 — 提出後に読む)

## この課題の狙い

**認証(Authentication)と認可(Authorization)の境目**を、体で覚える。
401 と 403 はどちらも「拒否」に見えるが、原因も対処もまったく違う。
ここを混同すると、実務では「キーを何度も再発行したのに直らない」
「権限申請すべきところでリトライ実装を入れる」という迷走が起きる。

## 報告書に入っていてほしい要素

### 何を試したか

- **同じ閲覧用キー**で、参照(GET・200)と作成(POST・403)を両方試したこと。
  この「同じキーで結果が分かれる」対照こそが、401 との決定的な違いを示す証拠
- 参照の成功を**先に**確認していること — 「認証は通っている」を自分で実証してから
  403 を観察する順序が、切り分けの型として正しい

### 403から何が分かったか

- エラーコードは `INSUFFICIENT_SCOPE`。**認証は通った上で**、この操作(通知の作成)の
  権限が無い、と言われている
- 403 が返ってきた時点で「キーは有効」が確定する(無効なら 401 で止まっているはず —
  認証→認可の評価順)
- リトライしても、キーの綴りを見直しても直らない。**直すのは権限の側**

### 401と403をどう切り分けるか

- 401 = 誰か分からない → キー・ヘッダーの側を疑う(付け忘れ・取り違え・形式)
- 403 = 誰かは分かるが権限が足りない → **権限の付与を依頼する**か、
  権限のあるキーに切り替える
- 監査ログでも切り分けられる: 401(キー無し)は suffix が null、
  403 は suffix が残った上で `INSUFFICIENT_SCOPE`(=有効なキーの権限不足)

## 発展(書けていたら実務レベル)

- 閲覧用キーに作成権限が無いのは発行ミスではなく**必要最小限の権限**という正しい設計。
  監視ツールが乗っ取られても、そのキーでは通知を偽造できない — 被害の上限を
  キーの権限で区切る、という考え方(C-002 の「rate limit は漏洩時の被害を抑える」と同系)
- 連携先への報告文まで書けていれば満点: 「閲覧用キーの動作を確認しました。
  参照は 200、作成は 403 INSUFFICIENT_SCOPE で、仕様どおりです」

## よくある不足

- 403 だけ見て、参照の成功(200)を確認しない — 「認証が通っている」証拠が無いと、
  401 との違いを実証したことにならない
- 「権限エラーが出た」で止まり、**次の一手**(権限を依頼する/キーを替える)まで
  書かない — 切り分けは対処につながって初めて意味がある
- 403 に対してリトライや再発行を対処として書いてしまう(→ 権限が変わらない限り直らない)
