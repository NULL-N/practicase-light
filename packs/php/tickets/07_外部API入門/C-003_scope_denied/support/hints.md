# C-003 ヒント(詰まったら上から順に)

## ヒント1: 閲覧用キーなのに 401 になる

2つの可能性を順に確認する。

1. **PCPサーバーが古い**: 閲覧用キーは最近追加されたもの。
   `docker compose up -d --build` で `pcp` を再ビルドしてから試し直す
2. キーの綴り: `PCP_TEST_KEY_readonly00003333` をそのままコピーする
   (`Bearer` の後の半角スペースも C-002 と同じ)

## ヒント2: 閲覧用キーの GET まで 403/401 になる

閲覧用キーは**参照はできる**設計。GET まで失敗するならスコープの問題ではなく、
ヘッダーの形式かキーの綴りの問題(ヒント1の2へ)。

## ヒント3: GET する通知IDが無い

C-001 / C-002 で自分が作成した通知のID(`ntf_...`)が使える。
分からなくなったら、フル権限キーで1件作成して、そのレスポンスの `id` を使う
(`support/spec.md` に作成のcurl例がある)。

## ヒント4: 403 が「エラーが出た」以上に読めない

レスポンスの `code` を見る。`INSUFFICIENT_SCOPE` = 「スコープ(権限)が足りない」。
このとき**認証は既に通っている**(通っていなければ 401 で止まっている)。
つまりサーバーは「あなたが誰かは分かった上で、この操作は許可していない」と言っている。

## ヒント5: 「切り分け」に何を書けばいいか

401 と 403 で「**次に何をするか**」がどう変わるかを書く。

- 401 → キー・ヘッダーの側を疑う(綴り・付け忘れ)
- 403 → キーは合っている。**権限の付与を依頼する**か、権限のあるキーに切り替える。
  綴りをいくら見直しても、リトライしても、403 は直らない

問い合わせを受けた側の視点で「401なら○○を確認してもらう、403なら○○する」と
書けると、実務の切り分け手順書に近づく。
