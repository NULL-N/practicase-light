# C-001 期待される報告の観点(答え合わせ用 — 提出後に読む)

## この課題の狙い

外部 API 連携の最初の一歩は「コードを書くこと」ではなく、
**仕様を読み、実際に呼び、返ってきた事実と残った記録を確認すること**。
C-001 はその一連の動作を、最小の API(通知1件)で一周する課題。

## 報告書に入っていてほしい要素

### 何を送ったか

- どのエンドポイントへ・どんな宛先/本文で・どのキー(=教材用テストキー)で送ったか
- `fail-` 宛先の通知も送ったこと(失敗系を自分で試す姿勢そのものが実務の型)

### 何が返ったか

- `201` と通知ID(`ntf_...`)・`status: queued`
- **同じ ID を時間を空けて GET すると結果が変わる**という観察
  (`queued` → `delivered`、`fail-` 宛先は `failed`)。
  「作成の成功」と「配送の結果」が別物である、という非同期 API の基本がここにある

### 監査ログに何が残ったか

- POST(201)や GET(200)が時刻付きで記録されていたこと
- **API キーは末尾4桁(`api_key_suffix`)だけ**が残り、全文はどこにもないこと。
  ログは流出しうる前提で設計する — 認証情報をログに平文で残すのは実務の典型的な事故
- `/v1/health` と `/v1/audit-log` 自身は記録されないこと(記録する価値のない
  定期アクセスでログを埋めない、という設計判断)

## よくある不足

- 通知IDを書き写さず「成功した」とだけ書く(→ 報告を受けた側が追跡できない。
  ID・時刻・状態は報告の最小セット)
- `queued` を「まだ失敗している状態」と誤解する(→ 受付済みと配送済みは別の概念)
- 監査ログの記述が「ログがあった」で終わる(→ 何が**残っていないか**まで見るのが監査ログの読み方)
