# T-002 段階ヒント

**運用**: 詰まってから開きます。1段読んだら閉じて、また15分は自分で手を動かしてください。

## Hint 1: まず考える方向

症状は「チェックを入れても絞り込まれない」。チェックボックスの値は、**画面 → パラメータ → SQL** の順に旅をします。
途中で消えているのか、届いているのに使われていないのか — 「どこまで届いているか」を順に確かめるのが、この調査の型です。
`docs/03_参考資料/code-tour.md` の「1つの操作がコードを流れる順番」で、検索がどの層を通るかを確認してください。

---

## Hint 2: 見るファイル

画面の入口は `packs/php/app/public/projects/index.php`。
`remote_only` がどう受け取られ、どのメソッドに**引数として渡されているか**を見ます(ここまでは正常か?)。
渡った先が `src/Repository/ProjectRepository.php`(ARC-2: SQL はこのクラスに集約)。
開いたら、`docs/01_設計資料/features.md` F-03 の「検索条件」の表を横に並べて照合します。

---

## Hint 3: 仕様と実装のズレ

F-03 の検索条件は **2つ**(キーワード / リモート可のみ)。
`search()` の中で、`$where` に条件を積んでいる if ブロックを数えてください。仕様の数だけあるか?
そして — 引数で受け取っているのに、メソッドの中で**一度も使われていない**ものは無いか?
(T-001 には「定義されているのに呼ばれていない」ものがいました。今回は「受け取ったのに使われていない」)

---

## Hint 4: check が見ている両側

`tests/checks/T-002.php` は開いて読んで構いません。この課題の合格条件そのものです。
見ているのは **ON と OFF の両側**: ON でリモート可の1件だけになるか、そして **OFF で2件とも出るか**
(F-03「OFF のときは絞り込まない」)。片側だけ確認して満足しないこと。
直すときは、既存の if ブロックと同じ「`$where` に積む」型に揃えます。新しい書き方を発明しないこと。

これ以上のヒント(修正コードそのもの)はありません。ここまで来たら、あとは書いて check を回すだけです。
