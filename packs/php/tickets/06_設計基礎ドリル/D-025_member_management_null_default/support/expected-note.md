# D-025 突き合わせ(模範との比較)

自分の `reports/D-025_null_default_note.md` と見比べてください。**理由の言葉づかいが
違っても構いません** — 「業務上、無いと困るか」という判断の軸が合っていれば、
この課題の目的は達成しています。

## 模範の内容(1つの妥当な例)

| テーブル | カラム | 判断 | 理由 |
|---|---|---|---|
| departments | name | NOT NULL | 部署名が無いと部署として機能しない |
| members | name | NOT NULL | 氏名が無いとメンバーとして特定できない |
| members | email | NOT NULL | 連絡手段として必須 |
| members | status | NOT NULL, DEFAULT '在籍中' | 新規登録時点では在籍中であるのが自然 |
| members | department_id | NOT NULL | 所属先の無いメンバーは想定しない |
| members | joined_at | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 登録された時点を自動で記録するのが自然 |
| members | left_at(退職日) | NULL可 | 在籍中のメンバーには退職日が存在しないため |

## 別解について

- `left_at`(退職日)のような任意カラムを追加しない案も妥当です(必須ではありません)。
  追加した場合は、なぜNULL可にしたかの理由が書かれていることが大事です
- DEFAULTの具体的な値の表記(`'在籍中'`か`'active'`か)は自由です

## 見つかりやすい抜け

- **department_idをNOT NULLにし忘れる**: 「所属部署が無いメンバー」を許すかどうかは
  実務上の判断ですが、この課題では「メンバーは必ずどこかの部署に属する」という前提で
  NOT NULLとします
- **状態にDEFAULTを検討しない**: 「NOT NULLだから何か値を入れなければいけない」ことは
  分かっても、「新規登録時点で自然な値は何か」まで考えられていないケース

## この課題で本当に鍛えたいこと

NULLを「エラーを避けるための機械的な設定」ではなく、「この情報は業務上いつ存在し、
いつ存在しないのか」という業務理解として説明できる力です。
