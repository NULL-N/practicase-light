# D-024 突き合わせ(模範との比較)

自分の `reports/D-024_column_types_note.md` と見比べてください。**型の具体的な表記
(TEXT/VARCHAR、INTEGER/INT等)が違っても構いません** — 「文字列/数値/日時/決まった値」の
どれに分類したかが妥当であれば、この課題の目的は達成しています。

## 模範の内容(1つの妥当な例)

| テーブル | カラム | 型 | 理由 |
|---|---|---|---|
| departments | id | 数値(INTEGER) | 行を一意に特定するための番号 |
| departments | name | 文字列(TEXT) | 部署名は決まった長さの無い言葉の情報 |
| members | id | 数値(INTEGER) | 行を一意に特定するための番号 |
| members | name | 文字列(TEXT) | 氏名も決まった長さの無い言葉の情報 |
| members | email | 文字列(TEXT) | メールアドレスも文字の情報 |
| members | status | 文字列(TEXT。決まった値のみならENUM相当でも可) | 「在籍中」「退職済み」など決まった種類の値を表すため |
| members | department_id | 数値(INTEGER) | departmentsテーブルのidを指し示す参照のため |
| members | joined_at | 日時(DATE) | 入社日という「いつ」の情報のため |

## 別解について

- `status`をENUM型で厳密に表現しても、シンプルに文字列型として扱っても、どちらも妥当です
  (「決まった種類の値」という分類が合っていることが大事で、実際の型表記は自由)
- `id`を数値ではなく文字列(UUID等)として設計する案も、理由が説明できていれば妥当です。
  ただしその場合も「文字列だが一意な識別子として使う」という理由が書かれているのが望ましいです

## 見つかりやすい抜け

- **理由を書かずに型名だけで終える**: 「name: 文字列」で止まってしまうケース。
  一言でよいので「なぜ」を添えてください
- **部署への参照を文字列(部署名そのもの)にしてしまう**: D-023で「部署への参照」とした
  意図(部署テーブルのidを指し示す)を思い出してください。参照は数値型にします

## この課題で本当に鍛えたいこと

「なんとなくTEXTでいいや」ではなく、情報の性質(言葉か・数か・日付か・決まった選択肢か)から
逆算して型を選ぶ習慣です。理由を言葉にできることが、後で他人に説明する力にも直結します。
