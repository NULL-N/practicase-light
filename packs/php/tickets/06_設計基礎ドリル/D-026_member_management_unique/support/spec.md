# D-026 spec: UNIQUE判断メモの型

## 提出物の型(reports/D-026_unique_note.md)

この節立てをそのまま使ってください。**テーブル名はこれまでと同じに**してください。
判断は「UNIQUEあり」「UNIQUEなし」のどちらかの言葉で明示し、区切りには全角ダッシュ
「—」を使います(この2つの言葉をそのまま使うのが確実です。表記ゆれがあると
判定できないことがあります)。

```markdown
# D-026 UNIQUE判断メモ

## テーブルごとのUNIQUE判断

### テーブル: <これまでのテーブル名>
UNIQUE判断:
- <カラム名>: UNIQUEあり — <理由>
- <カラム名>: UNIQUEなし — <理由>

### テーブル: <これまでのテーブル名>
UNIQUE判断:
- <カラム名>: UNIQUEあり — <理由>
- <カラム名>: UNIQUEなし — <理由>

(これまで洗い出したカラムの数だけ繰り返す)
```

## 判断の考え方

「このカラムの値が、2行で同じだったら、業務として困るか?」を自問してください。

- **メールアドレス**: 同じメールの人が2人いたら、どちらのメンバーか特定できない → UNIQUEあり
- **氏名**: 同姓同名は普通に存在する。氏名だけで「同一人物」と判定するのは危険
  → UNIQUEなし
- **状態**: 「在籍中」のメンバーは何人いても構わない → UNIQUEなし
- **部署名**: 会社によって方針が分かれます。同名部署を絶対に許さない運用なら
  UNIQUEあり、組織変更で一時的に同名部署が併存しうるならUNIQUEなし。**どちらでも
  正解ですが、理由が必要です**

## 決めなくてよいこと

- PRIMARY KEY・FOREIGN KEY(D-027の仕事)
- コード(この章では一切書かない)

これらに触れてしまっても check が落ちることはありませんが、この段階では
「重複してよいか」だけに集中してください。
