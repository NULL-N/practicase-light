# D-023 spec: カラム洗い出しメモの型

## 提出物の型(reports/D-023_columns_note.md)

この節立てをそのまま使ってください。D-022で決めたテーブルの数だけ「### テーブル: 」の
ブロックを繰り返します。**テーブル名はD-022で使ったものと同じにしてください**
(check がテーブル名から「部署にあたるテーブルか」「メンバーにあたるテーブルか」を
探すため)。

```markdown
# D-023 カラム洗い出しメモ

## テーブルごとのカラム候補

### テーブル: <D-022で決めたテーブル名>
カラム候補:
- <カラム名1>
- <カラム名2>
- <カラム名3>

### テーブル: <D-022で決めたテーブル名>
カラム候補:
- <カラム名1>
- <カラム名2>

(D-022で洗い出したテーブルの数だけ繰り返す)
```

## 考え方のヒント

「このテーブルの1行を、実際に人に説明するとしたら、何を伝えるか」を考えてみてください。
例えば部署なら「どの部署か分かるように、名前が要る」。メンバーなら「誰か分かるように名前、
連絡が取れるようにメール、今も在籍しているか分かるように状態、どこの部署か分かるように
部署への参照」といった具合です。

## 決めなくてよいこと

- 型(INTEGER・TEXT等。次のD-024の仕事)
- NULL許容・NOT NULL(D-025の仕事)
- UNIQUE(D-026の仕事)
- PRIMARY KEY・FOREIGN KEY(D-027の仕事)
- コード(この章では一切書かない)

型や制約まで書いてしまっても check が落ちることはありませんが、この段階では
「何の情報が要るか」だけに集中する方が、後の段階が考えやすくなります。
