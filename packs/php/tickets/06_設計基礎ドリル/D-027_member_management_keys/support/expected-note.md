# D-027 突き合わせ(模範との比較)

自分の `reports/D-027_keys_note.md` と見比べてください。**理由の言葉づかいが違っても
構いません** — 「主キーは変わらない識別子にする」「参照はFKで表す」という判断の軸が
合っていれば、この課題の目的は達成しています。

## 模範の内容(1つの妥当な例)

| テーブル | 項目 | 判断 | 理由 |
|---|---|---|---|
| departments | PRIMARY KEY | id | 各部署を一意に識別する、変わらない番号のため |
| members | PRIMARY KEY | id | 各メンバーを一意に識別する、変わらない番号のため |
| members | PRIMARY KEYにしないもの | email | メールアドレスは変更されることがあるため |
| members | PRIMARY KEYにしないもの | name | 同姓同名がありうる上、氏名も変わりうるため |
| members | FOREIGN KEY | department_id → departments.id | メンバーがどの部署に属するかを表すため |
| members | 削除時の挙動 | 所属メンバーがいる部署は削除できない | 削除するとメンバーが所属先を失うため |

## 別解について

- 削除時の挙動を「メンバーの`department_id`をNULLにする」(部署が消えたら未所属扱いにする)
  という判断にするのも妥当です。ただしその場合、D-025で`department_id`をNOT NULLに
  していたことと矛盾しないか、自分で見直してみてください
  (矛盾に気づけたなら、それ自体がこの基礎編で鍛えたい力です)

## 見つかりやすい抜け

- **UNIQUEだからという理由でメールを主キーにしてしまう**: D-026のUNIQUE判断と、
  この課題の主キー判断を混同しているケースです。UNIQUEな情報が必ずしも主キーに
  向くとは限りません
- **外部キーが「どのカラムを指すか」まで書かれていない**: 「department_idは外部キー」
  だけで終わらず、「departmentsテーブルのidを指す」まで書いてください
- **削除時の挙動に触れない**: 主キー・外部キーだけで満足してしまい、
  「削除しようとしたらどうなるか」を考え忘れるケースが多いです

## この課題で本当に鍛えたいこと

「一意であること」と「識別子として使うのにふさわしいこと」は別の性質だと理解する力です。
実務でも、メールアドレスやマイナンバーのような「意味を持つ値」を安易に主キーにして
後で困る、という失敗はよくあります。
